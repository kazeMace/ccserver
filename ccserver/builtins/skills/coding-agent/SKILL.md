# Coding Agent 使用指南

## 何时独立完成（不 spawn 子 Agent）

- 单文件修改、小于 50 行的改动
- 明确定义的 bug fix，影响范围已知
- 简单的重命名、格式化、注释补充
- 你能在 3 轮以内完成的任务

## 何时 spawn 子 Agent

满足以下任一条件时，通过 `Agent` 工具委托给子 Agent：

- 任务跨越 3 个以上文件，且逻辑相对独立
- 需要大量探索（搜索、阅读、分析）后才能动手
- 任务可以拆分为互相不依赖的并行子任务
- 需要保护主上下文窗口（子任务结果很大时）

## spawn 子 Agent 的标准写法

```
通过 Agent 工具 spawn 子 Agent 时，prompt 必须：
1. 自包含：包含所有必要上下文，子 Agent 是空白上下文
2. 明确目标：说清楚要做什么、验收标准是什么
3. 指定范围：告知相关文件路径、不要碰哪些文件
4. 指定验证方式：跑什么命令来确认完成（AST check / pytest / curl）
```

示例 prompt 结构：
```
背景：[项目简介，2 句话]
任务：[具体要做什么]
相关文件：[路径列表]
验收：[运行什么命令证明完成]
约束：[不要修改哪些文件，不要引入哪些依赖]
```

## 复杂任务分解原则

1. **先探索后动手**：不确定影响范围时，先用 Glob/Grep 搜索，再制定修改计划
2. **最小变更**：只改完成任务必须改的，不顺手重构
3. **改完就验证**：每个子任务完成后立即运行测试或 AST 检查
4. **同类问题一起修**：发现一个 bug 时扫描同模块是否有同类问题

## 常见错误

- ❌ 直接修改文件，没有先读文件当前内容
- ❌ 声称完成但没有贴出验证命令的输出
- ❌ 修改了 A 但没检查 B 是否受影响
- ❌ 在子 Agent prompt 里省略关键上下文（子 Agent 没有你的记忆）

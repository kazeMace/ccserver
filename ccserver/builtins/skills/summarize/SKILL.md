# 摘要规范

## 何时需要主动摘要

- 工具输出超过 2000 行，只保留关键信息
- 对话历史进入 compaction 前，主动整理关键决策
- 需要向子 Agent 传递大量上下文时，先压缩再传递
- 用户明确要求"总结一下"

## 代码摘要格式

```
文件：path/to/file.py（共 N 行）
职责：[一句话说明文件做什么]
关键类/函数：
  - ClassName：[作用]
  - function_name(params)：[作用，返回什么]
依赖：[直接依赖的外部模块或关键内部模块]
注意事项：[非显而易见的约束或陷阱]
```

## 错误/日志摘要格式

只保留：
1. 第一个错误（触发点）
2. 最后一个错误（最终状态）
3. 涉及当前任务的中间错误
4. 堆栈的关键帧（入口 + 出错行，中间层可省略）

## 对话/决策摘要格式

```
决策：[做了什么决定]
原因：[为什么这样决定，而不是另一种方案]
影响：[哪些文件/模块受影响]
待办：[还有哪些事没做]
```

## 工具输出截断规范

- Bash 输出超过 200 行：保留前 20 行 + 后 20 行 + "（已截断 N 行）"
- 文件列表超过 50 项：只列出与任务相关的条目
- JSON 输出：只展示与当前任务相关的字段，其余用 `...` 省略

## 禁止事项

- ❌ 把工具原始输出全部粘贴给用户（尤其是几百行的 log）
- ❌ 摘要时丢失错误信息（错误永远比成功信息更重要）
- ❌ 把"没有相关信息"摘要成"正常"（两者不等价）

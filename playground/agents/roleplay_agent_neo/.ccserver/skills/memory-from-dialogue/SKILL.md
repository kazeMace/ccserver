# memory-from-dialogue

从真实对话记录中提取目标角色的**事实性记忆**，写入 `persona_memory.md`。

**职责边界**：
- ✅ 负责：角色的生活事实（认识谁、发生了什么、对什么有什么态度、有什么习惯）
- ❌ 不负责：语言风格、说话方式、行为模式、fewshot 示例 → 这些由 clone-persona 的 Step 2-4 处理

---

## 输入格式

由 clone-persona T2 分批调用，每批对应一个临时文件：

```
Use the memory-from-dialogue skill.
target_role: <角色名>
session_dir: data/sessions/<session_id>/
dialogue_file: /tmp/clone_persona/batches/batch_NNN.txt
```

**第一步**：用 **Read 工具**读取 `dialogue_file` 指定的文件，获取原始聊天记录。不接受直接传入对话内容。

---

## 执行步骤

### Step 1：语义分段

真实聊天中，一个完整的"信息单元"往往分散在多条消息里，必须以**话题段落**为单位理解。

**分段规则（按优先级）**：

1. **时间断层**：两条消息间隔 > 30 分钟 → 强制切段
2. **话题引入信号**：出现以下信号词后接新内容 → 切段
   - 「你知道...不」「对了」「话说」「顺便问你」「诶」「对了说个事」
   - 对方突然引入此前未出现过的人名 / 地名 / 事件
3. **节奏变化**：从快速短句互动切换为一方连续多条输出 → 可能是话题转换，酌情切段

**每段输出**：
```
[段落 N] 时间范围：HH:MM — HH:MM
话题摘要：一句话描述这段在聊什么
内容：
  （原始消息，保持原样）
```

---

### Step 2：逐段提取角色的事实性记忆

对每个段落，以**整段对话**为上下文提取 `target_role` 的信息。

#### 2a. 归属验证 — 只提取角色自述

每条候选信息提取前，确认是否属于 `target_role` 的**第一人称自述**：

| 消息类型 | 示例 | 处理 |
|---------|------|------|
| 角色第一人称自述 | 「我今天好累」「我不喜欢他」 | ✅ 提取 |
| 角色描述对方 | 「你最近是不是很累」「你喜欢打游戏对吧」 | ❌ 跳过 |
| 角色转述对方说的 | 「你说你上次去了上海」 | ❌ 跳过 |
| 角色纯应答 | 「嗯」「好」「哈哈」「我懂」「行」 | ❌ 跳过 |
| 角色情绪感叹 | 「哇？」「我靠」「啊啊啊」 | ❌ 跳过（无事实信息） |
| 角色问对方问题 | 「你要去不」「你知道吗」 | ❌ 跳过 |

#### 2b. 跨消息语义重建

段落内信息需要**综合双方消息**才能完整理解——对方提供背景，角色的回应表明态度/立场。

**规则**：
- 不能只看角色单条回复，要结合对方的话理解语境
- 提取结果必须是完整的语义单元（不是碎片）

**示例**（角色=角角新）：

原始对话段落：
```
对方: 你知道小温要结婚不 这个月
角色: 哇？ 不知道哎 她没和我说
对方: 我和你讲 偷偷讲
对方: 17号
角色: 行 给份子钱行了
对方: 但是她前两天和我说，叫了tyy 我犯膈应……
角色: 我靠
对方: 然后又不好请假 我就说我不回去了 你别和林林说
角色: 行 给份子钱行了
角色: 我肯定不和她说
```

提取结果：
- 认识一个叫"小温"的人，关系不够亲密（对方结婚没有主动通知角色）
- 知晓小温将在10月17日结婚，本人不打算参加，打算随礼
- 对"tyy"这个人有负面感受（被提及时反应强烈）
- 答应帮对方保密，不告知"林林"

---

### Step 3：专有名词核实

提取完成后，扫描所有条目中的**专有名词**，判断是否需要联网搜索。

**需要核实的类型**：
- 人名缩写或代号（`tyy`、`小温`、`林林`）
- 游戏 / 影视 / 音乐名称
- 品牌 / 产品名
- 地名 / 场所名

**判断逻辑**：

```
对每个专有名词：

① 先在对话原文中确认归属 — 是角色提到的还是对方提到的？
   如果是对方提到、角色只是被动知情 → 记录关系即可，不展开

② 判断是否可搜索：
   - 人名/代号（tyy、林林）→ 无法搜索，记录上下文含义即可
     格式：「聊天时提到一个叫"xxx"的人，[上下文显示的关系/态度]」
   - 可能是公开内容（游戏/品牌/艺人/地点）→ 联网搜索确认

③ 需要联网时，调用 web-search agent：
   Use the web-search agent.
   用户消息：[专有名词] [上下文描述，如"是一个游戏"/"是一个品牌"]

   - 搜索确认名词真实存在、含义与上下文匹配 → 保留，用正确全称
   - 搜索结果显示名词不存在或语境不符 → 从提取结果中删除该条，不写入
   - 搜索结果不确定 → 保留但加注 `（含义存疑）`
```

**人名/代号类的处理示例**：
```
"tyy" → 无法搜索，上下文显示是令双方（至少对方）感到膈应的人
  写入：聊天时提到一个叫"tyy"的人，角色和对方对其有明显负面态度

"林林" → 无法搜索，上下文显示是共同认识的第三人、需要对其保密
  写入：聊天时提到一个叫"林林"的人，是共同认识的朋友
```

---

### Step 4：信息分类与写入

将核实后的信息分类：

| 信息类型 | 条件 | 写入格式 |
|---------|------|---------|
| 持久属性（稳定事实） | 出现 ≥ 2 次，如城市/职业/爱好/感情状态 | `- [聊天时] <内容>` |
| 时效性事件 | 具体发生的事、计划、态度 | `- [YYYY-MM-DD] <内容>` 或 `- [聊天时] <内容>` |
| 对某人的态度 | 反复出现 → 持久属性；仅一次 → 事件 | 同上 |

**时间标注规则**：
- 聊天记录中有明确日期 → 转换为 `[YYYY-MM-DD]`
- 无法确定 → `[聊天时]`

**写入前检查**（读取 `<session_dir>/persona_memory.md`）：
- 已有相同或高度相似条目 → 跳过，不重复写
- 新条目与现有条目矛盾 → 保留更具体的，删除模糊的
- 文件不存在 → 创建，首行写 `# 角色新设定\n\n`

追加到文件末尾。

---

### Step 5：汇报结果

```
[memory-from-dialogue 结果]
处理段落数：X 段
提取条目数：X 条（持久属性 X 条，时效性事件 X 条）
联网搜索：X 次（已确认 X / 删除 X / 存疑 X）
跳过（归属不属于角色 / 无事实信息）：X 条
写入路径：<session_dir>/persona_memory.md
```

---

## 注意

- **只提取事实，不分析风格** — 语言特征、句式习惯、行为模式不在本 skill 范围内
- **宁缺毋滥** — 宁可漏掉一条不确定的，也不要写入一条归属错误的
- **人名/代号不猜测** — 无法联网确认的人名，记录上下文关系即可，不展开推测
